Moving Your Company To Groovy

In 6 Examples and 12 Easy Steps

#1: Java Was Too Fat

  • The year is 2007, and fat Java web frameworks ruled the land. People had just started talking about Ajax (remember that?).
  • 3+ year investment In Java, which was itself a re-write of code written in 2001 (ported from classic ASP/VB).
  • This was my first large Java project but I had hit a wall, I was spending all of my time fixing bugs and very little time developing new features.
  • I tried a few of the cool frameworks at the time but found they didn't help much, they mainly just required that I write different kinds of boilerplate code.

#1: Go On A Groovy Diet

  • The main problem was Java. String handling sucked, RegEx support was complicated, type information made it hard to evolve the code, testing in Java was painful, ClassCastExceptions were common, verbose code etc.
  • Switching to Groovy gave me instant relief to all of these problems because I was able to evolve my code piecemeal...and this was before joint compilation
  • Every time I refactored from Java -> Groovy code size would decrease a factor of between two and five
  • This was the peak of Rails hype, but before Grails hit 1.0. I was tempted to jump to Rails, but knew that it would be a complete re-write that would take 6 months...and I didn't have 6 months to spare
  • Got back on track. Would develop for several more years and left when it became clear that there wasn't anything left to do

#2: Groovy Left a Bad Taste

  • Groovy dynamicity was abused, hurt by static type checking
  • People unwisely using Groovy features, such as closures, just because they could
  • Hard time upgrading off of Grails 1.x
  • Bad performance, though at this company nothing is ever performant enough

#2: Add Some Groovy Sugar and Rum

  • Every system performs well in different ways: optimize the right thing
  • Big data project that had to churn through terabytes of data in under 5 seconds
  • Data stored in a columnar store that was highly dependent on submitting the correct queries, select * from x; was NEVER the right answer
  • The queries were highly dynamic and had to be customized for each installation
  • Make a Groovy DSL, spend a few more microseconds generating the queries, then save several minutes in server side processing

#3: We Need a Little More Power

  • System supported 4 out of 24 possible network decoders
  • No JVM library supported all 24 decoders
  • Most didn't support full decodes, they were ad-hoc and partial
  • Speed was a constant issue. Flexibility always led to slower code

#3: Let Groovy do the Heavy Lifting

  • Groovy can perform well, sometimes even better than Java
  • I offered to implement all 24 decoders, support all fields, provide a flexible plugin system to support all extensions, have it perform better than all previous JVM based decoders, and to do it a couple of months
  • We delivered on all counts.
  • Because Groovy
  • @CompileStatic + AST Transformations allowed us to implement a compile time DSL. Implementing the decoders was literally copying the spec out of pdf files and massaging them a little inside Groovy files
  • Compile Time DSL also allowed custom code to be added at compile time, lazy or eager decoding with no additional code, and accessing of data as Maps or properties.

#4: The Scripts are a Little Tangled

  • Extremely brittle deployments, especially in DB changes.
  • On deploy day people brought their PL/SQL scripts, threw them in a pile, and expected the DBAs to sort them out
  • Script runs were sequence dependent
  • Needed different scripts to run in different environments
  • Oh yeah, solution better just be plain PL/SQL scripts so DBAs can examine them

#4: Gradle: The Detangler

  • Ordering dependencies should always suggest a build tool, that's what they are good at.
  • Solution: Custom gradle plugin written in Groovy
  • Built custom deployer on top of Gradle application plugin and using Flyway
  • DBAs still don't know what the guts are made of because they just see a zip file that contains a bat file and some PL/SQL scripts.
  • It contains other stuff too, but I don't point that out.

#5: The Critical App Crashes

  • Extremely mission critical app crashes for long periods of time
  • Simple re-boots are risky and in general do not help because the app is processing data real time, it just goes back to crashing.
  • Written in Java with lots of custom Neo4j extensions
  • When it was working correctly it took around 0.5 seconds to match a person to a survey, under bad conditions up to 5 seconds, under really bad conditions it would never return.

#5: Groovy Flavored Bugatti

  • Realize that Neo4j was the incorrect choice for this project and that it was mainly being used to analyze complex expressions
  • Implement custom expression DSL and an efficient in memory representation of the data, both in Groovy
  • Optimized datat access access layer written in Groovy (Duh!).
  • Completely replace system. Response times drop to 20 ms. Can exhaustively match a person to all available surveys in less than 1 ms.

#6: Immutably Slow

  • Application that computes person's data points is very critical
  • Written in Scala so computational algorithm starts out immutable
  • Eventually part of the code is written in a mutable style either for efficiency or because it was the only way people could get it done
  • Computes state in around 10 ms per person. While that sounds fast, how long to compute state for 6.5 million + people?
  • Real problem is the amount of garbage generated under heavy load leading to lots of time spent in GC and unbelievably frequent and long GC pauses.

#6: Groovier Immutablity

  • I want to be clear: Immutability is usually a clear win, I use final more than anyone I know
  • You will eventually have to mutate state, plan for this up front and do it intelligently
  • Solution: Yep, re-write in Groovy
  • New solution is probably more immutable than the previous solution, but we mutate when speed counts, which is accessing a very efficient primitives based Map
  • We changed the original tree based algorithm to a DAG based algorithm
  • Computes state in around 0.2 ms per person and generates ZERO garbage
  • Bonus: Found several long standing and unknown bugs because we expressed the solution in the cleanest and most natural way

Why Groovy?

It allows you to express the solution to your problem in the most natural manner possible because it gives you all the tools you need to do this

What Tools Do I Need?

  • Static AND Dynamic Typing
  • Runtime AND Compile Time Metaprogramming
  • OO AND Functional Programming (Mutable AND Immutable)
  • Objects AND Primitives
  • Low Level AND High Level Coding Styles
  • Run as Script, Dynamic Code Evaluation, OR Static Build
  • Operator Overloading
  • Language Extensibility
  • Ability to mix any or all of the above
  • Seamless integration with Java and the JVM
  • As far as I know, only Groovy fits the bill

So What Works?

Understand the Problem

  • Be as specific as possible in understanding the problem
  • Take as much time as you need for this step. Think a lot and engage in HDD, Hammock Driven Development. Get philosophical and computer sciencey
  • Bad Example: We have too much unreadable code
  • Good Example: We have too much unreadable code because we have implemented an ad-hoc interpreter for our configuration system. We need the ability to compile and execute code at runtime using a real language.
  • Bad Example: We have too much unreadable code
  • Good Example: Our code is unreadable because we have no unified and compact way of specifying the business logic. We need to pull the logic together into a DSL that clearly expresses what the business does, ideally in a single file. Even better if we can use that single file to produce a visual representation of business logic.

Pitch It

Now that you have identified the problem, it's time to match the tool you want to use to the identified problem. This rarely is going to be a formal pitch, I've only done that a single time. If the problem you are solving is real, causes pain, slows things down, and/or costs money people are usually pretty reasonable about possible solutions.

History

  • No organization is a blank slate. Knowing how you got here helps you to tailor the pitch and avoid re-treading the same ground
  • The quickest way to get shot down is when your boss says We already tried that. There usually isn't a way to come back from that unless you can show No we didn't try that.
  • Talk to people and listen to their war stories in the org. It builds camraderie, shows you care, and gives you valuable insights into history
  • Source code control systems are a valuable source of history
  • Don't just talk to developers. Also talk to managers (they were often developers), production control/dev ops, QA engineers, DBAs, and anyone else who knows something.
  • A huge part of therapy is empathy

Culture

  • Every organization has a unique culture, since you are already talking to everyone, you should have a good feel for the culture
  • The best attempts at change will match the organizations strenths and weaknesses
  • Cost Sensitive: Your proposed change should emphasize how this change cuts costs
  • Blame Culture: Assume the blame for what happens next with the tool you are proposing
  • Risk Averse: Emphasize that this is the least risky option
  • IMPORTANT, I am not suggesting you lie or manipulate. There are lots of ways to tell a story, lots of ways to sell something. You need to speak the language of the culture you are in. You don't sell minivans to moms on top speed and you don't sell Ferraris to rich young punks on safety features

Honesty

  • Be honest to yourself, you have to live with this decision and see it through to the end. Are you ready?
  • Your IDE may permanently or temporarily break with your new language, can you live with that?
  • Are you prepared for the learning curve or have you already traveled far enough up the curve that you are confident it won't be a problem?
  • This may be harder to debug, but you will debug less, can you live with that?
  • This will be easier to test, but will you actually test more?
  • What happens if this fails? Are you prepared for this? Do you have a contingency plan?
  • Have you identified the problem and does this solve it?

This Isn't Playtime

If your boss thinks that you are just exploring stuff willy-nilly or are engaged in RDD, Resume Driven Development, you will lose or your new approach will be cut short.

Persistence

  • One loss is not permanent
  • One victory is not permanent (see examples)
  • It's going to take months or even years for a permanent victory

Play to the JVM

  • The risk of implementing something in an alternative JVM language is really quite small; Groovy, Scala, Kotlin, and Clojure all have exit paths
  • Are you already using one of these languages as a library and don't even realize it? Running "gradle dependencies" might be surprising

Low Risk == Low Rewards

  • The easiest way to sell Groovy is by selling
    1. Better builds with gradle
    2. Better testing with Spock
  • Unfortunately testing and builds are usually not seen as big money makers or critical to success, so showing improvement here is not a big selling point.
  • For me, the best way to use a new technology is to solve hard problems with the new technology.

Confidence

  • If you don't show confidence, nobody is going to be confident in you or the new technology
  • Write a proof of concept on your own initiative and demo it
  • Get confident: show up at user groups, give a talk on the new tech, contribute to open source, start a project, etc.

Have Fun

  • Fun is infectious and you are more likely to infect everyone with a new technology than you are going to convince them (people just aren't that rational, including engineers).
  • Even if everything fails, at least you had fun.

It's Up To You!

  • If you have listened to this you certainly care about software engineering.
  • You are almost certainly one of the top engineers where you work
  • Because of this, if anyone has a shot of doing this, it's you
  • Now go make it happen!
